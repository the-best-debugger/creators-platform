# Postman Guide

This guide explains how to set up Postman for this project, import the collection and environment, and run the API tests in the recommended order.

## 1. Setup

- Install Postman (https://www.postman.com/downloads/).
- Start your backend and frontend so the API is available. Example (project root):

```bash
# using docker-compose (if configured)
docker-compose up --build

# or run services manually in two terminals
# Terminal 1
cd server
npm install
npm run dev

# Terminal 2
cd client
npm install
npm run dev
```

- Determine the API base URL (default `http://localhost:5000`).

## 2. Import Collection & Environment

1. Open Postman → Click `Import` → `File` and choose the collection JSON file provided (`Creator's Platform API.postman_collection.json`).
2. Import the environment JSON (`Local Development.postman_environment.json`) the same way.
3. Open the imported environment and set `baseURL` if it is not correct (e.g. `http://localhost:5000`).

Note: If you don't have the exported JSON, you can create requests manually following the request details in the collection.

## 3. Important Environment Variables

- `baseURL` — The API base URL used by requests. Example: `http://localhost:5000`.
- `authToken` — JWT token saved after `Login` (or from `Register` auto-login). Used in `Authorization` header: `Bearer {{authToken}}`.
- `createdPostId` — Saved by the `Create Post` test script; used by `Update` and `Delete` requests.
- `firstPostId` — Optionally saved when fetching posts.
- `timestamp` and `randomInt` — Helper values generated by pre-request scripts for unique titles/emails.
- `currentUserId`, `currentUserName` — Optionally saved by the `Login` script.

Make sure the active environment in Postman contains the correct `baseURL` before running authenticated requests.

## 4. Authorization Header

For authenticated requests add the header:

```
Authorization: Bearer {{authToken}}
```

You can add this header at the collection level so every request inherits it.

## 5. Order of Running Requests (recommended)

1. **Health Check** — quick way to confirm the server is up: `GET {{baseURL}}/api/health`.
2. **Register User** — `POST {{baseURL}}/api/users/register`.
   - Tests will attempt to auto-login and save `authToken`.
3. **Login** — `POST {{baseURL}}/api/auth/login` (if you skipped auto-login).
   - Confirms credentials and stores `authToken`.
4. **Create Post** — `POST {{baseURL}}/api/posts` (requires `authToken`).
   - Tests save `createdPostId`.
5. **Get All Posts** — `GET {{baseURL}}/api/posts?page=1&limit=10` (validates data and may save `firstPostId`).
6. **Update Post** — `PUT {{baseURL}}/api/posts/{{createdPostId}}` (uses `createdPostId`).
7. **Delete Post** — `DELETE {{baseURL}}/api/posts/{{createdPostId}}`.

Run requests individually while developing, or use Postman's Collection Runner to run the sequence.

## 6. Tests & Scripts

- Each request includes a **Tests** script (paste into the request Tests tab) that asserts status codes and response shapes.
- `Register` has a test script that will auto-call `Login` and save `authToken` to the environment.
- `Login` test saves `authToken` and optionally user info.
- `Create Post` test saves `createdPostId` for downstream tests.

## 7. Running the Collection Runner

1. Click the collection → `Run` → Collection Runner opens.
2. Choose the environment, set iteration count (usually 1), and click `Start Run`.
3. Monitor results and inspect failures in the runner summary.

## 8. Troubleshooting

- If tests fail with authentication errors, ensure `authToken` is set in the active environment.
- If socket/real-time tests are involved, ensure the client has connected and server logs show socket connections.
- Check server logs (terminal running `server`) for errors.

## 9. Notes

- Adjust request bodies to match your desired test data. Use `{{timestamp}}` and `{{randomInt}}` to avoid collisions.
- If you prefer Register to return a token directly, enable the server change that returns a JWT on registration (ask me to implement it).
